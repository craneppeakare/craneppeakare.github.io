I Built a Scrabble Trainer to Beat My Friends

So me and my close friends love playing Discord’s in-client game Letter League, which is based off of the popular word game Scrabble. We’d play for hours at a time and one of our friends in particular, we will call him “Nigel”, tends to consistently win somewhat often. And I’m an ultra competitive person so I built a trainer program to help me find 7 letter words, known as “bingos” in scabble, and go on the training arc of my life and beat him to the ground.

Planning and Design

I used a Trello board to keep track of essential features and functions for the prototype as well as store any optional QOL features that I may want to add later down the line.

Trello boards are my favourite because it frees up so much head space so you can focus on one task at a time without ever feeling like you’re fogetting something. They are also blazingly fast to setup and easy to use. Although I’ve used options such as Jira and Notion, Trello just works straight away and is my preffered method of project organization for personal projects.

After lining up all the essential features and sketchboarding what the UI would generally look like In my head, I began implementing it. I also took special inspiration from online word games like Wordle and aimed to design a minimal and sleek look to the game.

Grabbing the wordlist

For this project we will need to source a wordlist of 7-letter words. After some googling I was able to find one at wordfind.com. So I tried to webscrap it using python, selenium, and beautifulSoup4 but the web crawler actually got blocked so there goes that idea. That killed the mood but it doesn’t stop me from manually going to the source html and downloading a copy of the page to put into beautifulSoup4. Which is exactly what I did. It’s not a pretty solution but hey, it worked.

import bs4 as bs

def main():
    f = open("html.html", "r")
    sauce = f.read()
    f.close()
    page = bs.BeautifulSoup(sauce, 'html.parser')
    
    words = page.findAll('li', class_='dl')
    wordlist = ""
    for word in words:
        wordlist += "\"" + word.a.text + "\",\n"
    
    f = open("wordlist.tsx", "w")
    f.write("export default const Wordlist = [" + wordlist + "]")
    f.close()

And we make sure to write it as a tsx file and export it as an array. Now it’s as easy as drag and dropping the file into our project then we can include it.

Now to simply implement each of the rack buttons to move tiles from one rack to the other. This uses react states so we’ll need to make a shallow copy of the current rack and apply the changes then re-set the state to the new values. The idx is used to know which tile was clicked on with the left-most tile starting with 0. You can see how we pass this information in the Rack component above. Next we’ll implement these 4 buttons for basic functionality; “GIVE UP” to reveal the answer and get a new word, “CLEAR” to remove all tiles from the top rack to the bottom, “SHUFFLE” to shuffle the tiles on the bottom rack, and “SUBMIT” to test if the word formed by the upper rack is valid.

We’ll start with CLEAR and SHUFFLE since they are the easiest.

CLEAR will simply make a copy of the top rack removing any null tiles, then for each tile will add it back to the bottom tile in order. The same order if you would click each tile manually from left to right.

For SHUFFLE, there are many ways to shuffle an array of data, but for this solution I wanted to use a bit packing solution because it’s cool. To do so, we make a new empty rack and setup a bit packing variable picked to keep track of what index already has a tile placed into it. for each tile, we randomly generate a number from 0 to the length of the rack, called i, and if the i-th bit in picked is a 1 then we push that tile into the i-th index in our new rack and set the bit to 0, using ^= 1 << i, indicating that a tile has been selected for it’s place.

function clearSelection() {
  let sRack = selectionState.filter((t) => t !== null);
  let newRack = [...rackState];
  for (const t of sRack) {
    const i = newRack.indexOf(null);
    newRack[i] = t;
    if (t?.isBlank) newRack[i]!.letter = " ";
  }
  setRackState(newRack);
  setSelectionState(Array(configs.handSize).fill(null));
}

function shuffleRack(rack: (TTile | null)[]): (TTile | null)[] {
  let newRack: (TTile | null)[] = [];
  let picked = 0xff;

  for (let j = 0; j < rack.length; j++) {
    let i = Math.floor(Math.random() * rack.length);
    while (((picked >> i) & 1) === 0)
      i = Math.floor(Math.random() * rack.length);
    newRack.push(rack[i]);
    picked ^= 1 << i;
  }

  return newRack;
}

Now for the GIVE UP button, we open up a Mui Modal Button containing the answer and we generate a new problem for the user using the following function below:

import { Wordlist as WordfindList } from "../Utilities/WordfindWordlist";

function getNewWord() {
  let answers: string[] = [];
  let newRack: (TTile | null)[] = [];

  let wordlist = WordfindList;

  while (answers.length <= 0) {
    newRack = [];
    let tilebag = [...TileBag];
    for (let i = 0; i < configs.handSize; i++) {
      let rand = Math.floor(Math.random() * TileBag.length);
      if (configs.blankTilesOn === false && tilebag[rand] === " ") {
        i--;
        continue;
      }
      const l = tilebag.splice(rand, 1)[0];
      newRack.push({ letter: l, isBlank: l === " " });
    }

    answers = findAllAnswers(
      newRack.map((t) => t?.letter).join(""),
      wordlist
    );
  }

  setAnswerKeyState(answers);
  setSelectionState(Array(configs.handSize).fill(null));
  setRackState(shuffleRack(newRack));
}

For this we will need to update the AnswerKey state, the upper rack (selection) state, and the lower rack state. We begin by making variables for the answer key and bottom rack states. In the demo, you have the option to switch to a different wordlist from the settings page but for simplicity in this example we can set wordlist = WordfindList by default.

The while loop is where the real magic happens. There are two approaches: (1) simulate drawing tiles from a bag with the same frequencies seen in a traditional scrabble game, and (2) choose a random word from the wordlist.

  • (1) Is great at exposing the user to 7-letter words that they would normally encounter in a scrabble game. More practice with common words means it’s more effective practice to be looking out for them. But is a lot slower compared to approach 2.
  • (2) Very simple and blazingly fast. Gets the job done. However, can artificially make it harder the user as rare words are just as likely to appear as common ones.

In this example we go with approach (1) but in the real demo I have added a switch in the settings page that allows you to switch between the 2 approaches.

The SUBMIT button is fairly straight-forward. Just check if the top rack forms a valid word that exists in the wordlist. But the small caveat is that there can also be multiple solutions; multiple anagrams for the same set of tiles. So we need to keep track of all possible solutions. Plus, things get a whole lot more complicated with the addition of blank tiles. We call the function findAllAnswers() with the selected word to get all possible answers.

export function findAllAnswers(word: string, wordlist: string[]): string[] {
  let res = [];
  let w1 = word.split("").sort();

  for (let word2 of wordlist) {
    let w2 = word2.toUpperCase().split("").sort();
    if (w1.indexOf(" ") === -1 && w1.join("") === w2.join("")) {
      res.push(word2.toUpperCase());
    }

    w1.forEach((c1, i) => {
      if (w2.indexOf(c1) !== -1) w2.splice(w2.indexOf(c1), 1);
    });
    if (w2.length === 2 && w1[1] === " ") res.push(word2.toUpperCase());
    if (w2.length === 1 && w1[0] === " ") res.push(word2.toUpperCase());
  }
  return res;
}

We sort the given word then iterate through the wordlist, sorting each word in the wordlist as well. If our given word does not contain a blank tile, and the two words match then we add it to our result. Otherwise, we iterate on each character in our given word and remove the same character (if found) on the wordlist word. We then check if all that remains is a blank tile or two blank tiles. If so, then it is a possible solution and we add it to the results.

This may look O(n^3 logn) at first glance, (nlogn for sorting, n for iterating over wordlist, and n for looping after each character), but if we realise that each word is guarantee’d to be no greater than 7 characters then sorting and looping over each character is constant time. In other words, since word length is fixed the runtime with respect to it will not scale exponentially. The only variable is the size of the wordlist so we end up with O(n) time complexity.

Now with all that the final check to see if the final word is in the answer key should be straightforward. We just perform some checks, then notify the user if they got it right or wrong. The winning/invalid modal window is responsible for calling getNewWord() when it gets closed.

function submitSelection() {
  if (submission.some((t) => t == null)) {
    // Open a modal window telling user to use all 7 tiles before submitting
    setUseTilesModalOpen(true);  
    return;
  }
  let s = submission.map((t) => t!.letter).join("");
  if (answerkey.indexOf(s) !== -1) {
    // Open a modal window telling user that they found the bingo
    setWinModalOpen(true);  // Modal calls getNewWord() on close
  } else {
    // Open a modal window telling user that their submission was incorrect
    setInvalidModalOpen(true);
  }
}

In the demo version there are further checks because I implemented a limited number of tries. When the user fails to find the bingo in the limited number of tries it will automatically tell the user the right answer(s) and generate a new problem.
